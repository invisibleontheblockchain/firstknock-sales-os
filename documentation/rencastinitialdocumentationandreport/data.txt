2. Endpoint Comparison: /properties vs /listings/sale
The two endpoints draw from fundamentally different data sources, operate on different update
cadences, and answer different questions. Selecting the wrong endpoint does not merely reduce
accuracy — it produces a dataset that is structurally incapable of confirming a sale occurred.
2.1 Data Sources
The /properties endpoint aggregates data from three official public record sources: county Tax
Assessor records, Deed records, and Property Tax data. These are legally binding documents filed
with the county clerk upon the completion of a real estate transaction. Because a deed transfer is a
legal requirement of property ownership, every record in this dataset represents a transaction that
definitively occurred.
The /listings/sale endpoint aggregates data from the Multiple Listing Service (MLS), a cooperative
database maintained by real estate brokers. MLS data reflects listing activity — when a property is
marketed for sale — not transactional activity. A listing can be created, modified, and removed
without a sale ever occurring.

3.2 The daysOld Parameter Trap
The daysOld parameter on the /listings/sale endpoint measures the number of days since the
property was listed on the market, referencing the listedDate field. It does not measure the sale
date, the removal date, or any other transactional event. This creates two compounding failure
modes for FirstKnock's use case:
• False Negatives — Missed Sales: A property listed 400 days ago that sold last week will
NOT appear in a daysOld=90 query, because its listedDate is 400 days old. FirstKnock would
miss this confirmed sale entirely.
• False Positives — Non-Sales Included: A property listed 30 days ago that was quickly
cancelled or withdrawn WILL appear in a daysOld=90 query, because its listedDate is recent.
FirstKnock would waste resources targeting a property that never sold.
• No Remedy Within the Endpoint: There is no parameter combination within /listings/sale that
resolves this structural flaw. The endpoint simply does not contain the data required to
confirm a sale.
3.3 Why /properties saleDateRange is the Correct Filter
The saleDateRange parameter on the /properties endpoint filters directly on the lastSaleDate field,
which is derived from the official county deed record. This field is only populated when a deed
transfer has been legally recorded by the county clerk. A property cannot appear in a
saleDateRange query unless a confirmed sale transaction exists in the public record. This is the
only mechanism in the RentCast API that provides this guarantee.
The saleDateRange parameter accepts a single integer value (treated as a maximum range, e.g.,
saleDateRange=180 returns properties with lastSaleDate within the last 180 days) or a min:max
range format (e.g., 30:270 for properties sold between 30 and 270 days ago). For FirstKnock's
rolling lead generation use case, the single-value format with saleDateRange=180 is the
recommended configuration.

5.2 Why saleDateRange=180 is the Correct Window
To reliably capture homes that closed within the last 90 days, the saleDateRange must be set to
180. This is because a home that closed exactly 90 days ago may have had its deed recorded only

ago but took the full 90-day maximum lag to be recorded. The practical effect is that a 180-day
query reliably surfaces homes that actually closed within the last 90 days, which is FirstKnock's
true target window for fresh, actionable leads.

5.4 Practical Implications for Lead Freshness
FirstKnock should not interpret the county recording lag as a lead quality problem. A homebuyer
who closed 60 days ago and whose deed was just recorded this week is still a highly actionable
lead — they are newly settled, may not have received home services outreach yet, and the
owner.names and ownerOccupied fields provide the targeting precision needed for door-knocking
campaigns. The lag shifts the timing of lead discovery, not the quality of the lead itself.
OPERATIONAL INSIGHT: Because of the rolling nature of deed recording, running the same
saleDateRange=180 query every week will surface new leads each cycle — homes that closed
weeks ago but whose deeds were only just recorded by the county. A weekly refresh cadence is
essential to capture this continuous stream of newly recorded transactions.

7.4 De-duplication Strategy
When querying multiple adjacent ZIP codes, boundary overlaps or API data anomalies can
occasionally produce duplicate records. Use the following de-duplication logic to maintain a clean
database:
• Primary unique identifier: Use the formattedAddress field as the primary deduplication key. If
the native property id field is present at the root of the response object, use it as the
preferred identifier.
• Upsert logic: On each weekly refresh, check if the formattedAddress already exists in the
database before inserting. If it exists, compare lastSaleDate values and update only if the
incoming record is newer.
• Avoid radius queries for deduplication simplicity: Lat/lon radius queries introduce geographic
overlap that complicates deduplication. Sticking to exact ZIP codes (88001, 88005, 88012)
makes application-side deduplication straightforward and deterministic.
8. Limitations and Edge Cases
8.1 Deed Recording Lag — Very Recent Sales Are Invisible
The most significant operational limitation of the /properties endpoint is that homes closed within
the last 30–90 days will not appear in any query until their deeds are officially recorded by the
county clerk. There is no workaround within the /properties endpoint for this limitation. FirstKnock
must accept that the lead pipeline will always reflect a delayed snapshot of the market, and design
outreach workflows accordingly. The weekly refresh cadence described in Section 7 is the correct
mitigation — it continuously surfaces newly recorded deeds as they propagate through county
systems.

8.3 Non-Arm's Length Transactions
Not every deed-recorded transaction represents a traditional market sale. The lastSalePrice field
should be inspected for anomalous values that indicate non-arm's length transfers. A sale price of
$0, $10, or $100 typically indicates a property transfer into a family trust, an LLC, or between
family members — not a market transaction. These records should be filtered out of FirstKnock's
lead pipeline as the new 'owner' is unlikely to be a traditional homebuyer.
8.4 Corporate and Institutional Ownership
The owner.names array may contain corporate entity names rather than individual names. Strings
containing 'LLC', 'Trust', 'Bank', 'Inc', 'Holdings', or 'Properties' indicate institutional or corporate
ownership. These records should be flagged and routed to a separate workflow unless
FirstKnock's strategy specifically targets renters of corporate-owned homes, in which case the
property address (not the owner mailing address) is the relevant outreach target.
8.5 Geographic Coverage Gaps
The three Las Cruces ZIP codes (88001, 88005, 88012) cover the primary urban and suburban
areas of the city. However, some properties on the geographic boundaries of these ZIP codes may
fall into adjacent ZIP codes not included in the query set. If coverage gaps are identified during
field operations, add the relevant adjacent ZIP codes to the weekly query batch. Do not use a
single large radius query as a substitute — radius queries pull in rural and agricultural properties

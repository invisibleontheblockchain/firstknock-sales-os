API Drift Risk: 500+ Fragmented MLS Boards

The US real estate market operates across 500+ independent MLS boards, each with its own data schema, field naming conventions, and update protocols. Even aggregators with broad MLS coverage cannot guarantee schema consistency across all boards. This creates 'API drift' — a condition where field values, status codes, or data structures shift over time as individual MLS boards update their systems, causing silent validation failures in downstream applications.

RESO Web API Transition (April 2025 Deadline): The mandatory migration from legacy RETS to RESO Web API standard reduces integration time by 92% and standardizes JSON payloads. However, during the transition window, boards mid-migration may produce inconsistent outputs. Monitor provider changelogs closely through Q2 2025.

· Mitigation: Use providers (BatchData, ATTOM) that abstract MLS schema normalization internally

· Mitigation: Implement field-level validation in your ingestion layer — never assume field presence

· Mitigation: Set up automated schema drift alerts on critical fields (listing_status, list_date, close_date)

· Mitigation: Maintain a provider changelog subscription and schedule quarterly API contract reviews

Recommended Pre-Processing Layer: Smarty Streets

For any pipeline processing 1,000+ addresses per month, a dedicated address normalization pre-processing layer is strongly recommended before any property data API call. Impact on 7–14 Day Cache TTL Strategy

· MLS-Direct Providers (ATTOM, BatchData): Daily or sub-daily refresh cycles mean that within a 7–14 day TTL window, the source data will have been refreshed 7–14+ times. Staleness risk is low — the primary risk is the delta between the last cache write and the next MLS update, not the TTL duration itself.

· Aggregator Pipelines (RentCast, RevAddress): Refresh cadence is less transparent. Before committing to a 14-day TTL with these providers, contractually confirm minimum refresh frequency. A provider refreshing weekly would mean cached data could be up to 21 days stale at TTL expiry.

· Web-Crawl Providers (Datafiniti): Unsuitable for any caching strategy. With only 4% of new MLS listings captured within 24 hours, a 7-day cache of Datafiniti data would be structurally unreliable for active listing status.

· Deed-Based Providers (Estated): The 30–90 day deed recording lag exceeds the 14-day TTL window entirely. These providers cannot be used for active listing status validation regardless of cache strategy.

· Recommended TTL by Data Type: Set shorter TTLs (3–5 days) for active/pending listing status fields, which change frequently. Longer TTLs (10–14 days) are acceptable for stable fields like property characteristics, ownership records, and parcel data.

Adaptive TTL Strategy: Consider implementing confidence-weighted TTLs — properties in high-velocity markets (e.g., urban cores with <7-day average days-on-market) should use shorter TTLs (3–5 days) to reduce the risk of serving stale active/pending status. Properties in slower markets can safely use the full 14-day TTL.

Routing Architecture for 45-Second SLA

7. Input Ingestion: Receive address batch (up to 500 addresses). Log batch ID, timestamp, and source for audit trail.

8. Pre-Processing — Smarty Streets Normalization: Submit full batch to Smarty Streets batch endpoint. Receive standardized, CASS-certified addresses. Deduplicate normalized outputs. Flag unresolvable addresses for manual review queue.

9. Cache Lookup: Query local cache store for each normalized address. Return cached results for addresses with valid TTL (< 14 days). Route cache-miss addresses to primary API call.

10. Primary API Call — BatchData: Submit cache-miss addresses as a single bulk request (up to 2,000/request). Receive property status results. Target: sub-second response for ≤500 addresses.

11. Result Validation & Anomaly Detection: Validate response fields for schema completeness. Flag anomalies (missing status, unexpected null fields, schema drift indicators). Route flagged records to RentCast secondary verification.

12. Secondary Verification — RentCast (Anomalies Only): Submit anomaly-flagged addresses to RentCast for cross-validation. Reconcile conflicting results using BatchData as authoritative source.

13. Cache Write: Write validated results to cache store with TTL assignment. Apply adaptive TTL: 3–5 days for active/pending status; 10–14 days for stable property characteristics. Log cache write with provider source, timestamp, and TTL expiry.

14. Response Assembly: Merge cached results + fresh API results. Return complete batch response to calling application. Total target latency: <45 seconds end-to-end including normalization.

Cache Layer Design for 7–14 Day TTL with ToS Compliance

· Cache Store: Redis or equivalent in-memory store with persistent backup. Key structure: normalized_address_hash → {provider, data, written_at, ttl_expiry, data_type}

· TTL Enforcement: Hard expiry at 14 days maximum. Implement automated TTL enforcement at the cache layer — do not rely on application-level TTL checks alone.

· Data Type Segmentation: Separate cache namespaces for MLS-origin data (strictest TTL) vs. assessor/public record data (longer TTL permissible). Enables per-namespace TTL policy aligned with provider ToS.

· Audit Logging: Log all cache writes, reads, and deletions with timestamps. Maintain 90-day audit log for compliance documentation. Required for demonstrating ToS compliance during provider audits.

· Deletion Protocol: Implement automated data deletion at TTL expiry — not just expiry flags. Purge expired records from persistent backup within 24 hours of TTL expiry.

· Commercial License Prerequisite: Do not deploy cache layer to production until Commercial License Agreements with BatchData (and any other providers) are executed and explicitly grant 7–14 day internal caching rights.

Address Normalization Pre-Processing Pipeline

The normalization pipeline should be implemented as a stateless microservice sitting between the input ingestion layer and the cache/API routing layer. This enables independent scaling, versioning, and replacement of the normalization provider without modifying the core validation pipeline.

· Input: Raw address strings from calling application (any format)

· Step 1 — Format Parsing: Extract address components (street number, street name, unit, city, state, ZIP) using regex + rule-based parser

· Step 2 — Smarty Streets CASS Validation: Submit parsed components to Smarty Streets batch endpoint; receive standardized USPS-format output

· Step 3 — Deduplication: Hash normalized addresses; remove duplicates before downstream API calls

· Step 4 — Unresolvable Flagging: Addresses that fail CASS validation are flagged and routed to manual review queue — not passed to property API

· Output: Deduplicated, CASS-certified address list ready for cache lookup and API routing

Bottom Line Up Front

The definitive engineering recommendation for our high-velocity property validation pipeline is to implement BatchData as the primary routing engine, supported by RentCast for anomaly verification, and Apify / Bright Data as an absolute last-resort fallback for data gaps [1]. BatchData’s direct MLS integrations guarantee the necessary data freshness for listing statuses, while its bulk endpoint architecture easily clears the 500-address / 45-second latency threshold [1]. Recommendation #1 — Primary Provider: BatchData

Why #1





Data freshness priority: Relies on daily MLS feeds rather than deed-based public records, effectively bypassing the 30–90 day lag typical of county recordings [1]. Currently, 95% of top real estate systems update via MLS within 24 hours (with some refreshing every 15 minutes) [1].



Batch execution: Supports up to 2,000 results per paginated request, comfortably clearing our 100–500 address requirement in a single API call [1].



ToS compliance posture: Strict out-of-the-box restrictions against storage exist; local 7–14 day caching is only legally permissible via a negotiated Custom Commercial License Agreement [1].



Cost efficiency: Highly scalable pricing structure that aggressively discounts high-volume boolean status checks, bringing unit economics down to $0.031/record at peak volume sizes [1].

Integration Approach





API endpoint pattern: Utilize the dedicated BatchData bulk verification endpoint over standard single-record GET requests [1].



Authentication method: Secure integration via custom commercial API keys linked to our specific negotiated caching terms [1].



Batch request structure: Construct maximum-payload JSON payloads (batching 500 addresses per query) to minimize network round-trips and maximize the 99.99% uptime SLA [1].



Address normalization pre-processing step: Route all raw inputs through BatchData’s built-in USPS address normalization first to deduplicate queries before executing the validation logic [1].

Performance Projection





Expected latency for 100-address batch: Sub-second execution for the entire array payload [1].

-- ## SECTION 2: CORE CACHE TABLE — property_cache

CREATE TABLE property_cache (
    -- id: Native PG13+ UUID generation, avoids sequence bottlenecks in high-write workloads.
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    -- address_hash: SHA-256 of normalized address for O(1) dedup lookup. 
    -- WHY: String comparison on long addresses is slow; hashes provide fixed-length, fast index lookups.
    address_hash TEXT NOT NULL,

    -- normalized_address: lowercased, whitespace-collapsed, standardized.
    normalized_address TEXT NOT NULL,

    -- raw_address: original input as received for auditing and debugging.
    raw_address TEXT,

    provider_id TEXT NOT NULL,

    -- status: property_status enum NOT NULL.
    status property_status NOT NULL,

    -- location: PostGIS point, WGS84.

    -- latitude/longitude: denormalized for non-spatial queries.
    -- WHY: Avoids ST_X/ST_Y function overhead when exporting data to non-GIS downstream systems.
    latitude NUMERIC(10,7),
    longitude NUMERIC(10,7),

    -- raw_response: full API response blob.
    -- WHY: JSONB allows flexible schema evolution if providers change their payload structures.
    raw_response JSONB,

    -- cache_hit_count: tracks utility of the cache entry.
    cache_hit_count INTEGER DEFAULT 0 NOT NULL,

    -- last_accessed: tracks when the cache was last utilized.
    last_accessed TIMESTAMPTZ,

    -- created_at/updated_at: standard audit timestamps.
    created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL,
    updated_at TIMESTAMPTZ DEFAULT NOW() NOT NULL,

    -- expires_at: computed as created_at + TTL interval.
    expires_at TIMESTAMPTZ NOT NULL,

    -- is_stale: soft-flag for invalidation.
    -- WHY: Soft deletes/invalidations prevent locking issues during high-velocity reads.
    is_stale BOOLEAN DEFAULT FALSE NOT NULL,

    -- confidence_score: **0.000**–**1.000**, data quality signal.
    confidence_score NUMERIC(4,3),

-- ## SECTION 6: TTL INVALIDATION FUNCTION

-- WHY: Encapsulating this in a PL/pgSQL function ensures atomicity. 
-- The CTE updates the cache and queues the refresh in a single transaction, preventing race conditions.
CREATE OR REPLACE FUNCTION invalidate_stale_cache() 
RETURNS INTEGER AS $$
DECLARE
    invalidated_count INTEGER;
BEGIN
    WITH updated_cache AS (
        UPDATE property_cache
        SET is_stale = TRUE,
            updated_at = NOW()
        WHERE expires_at < NOW() 
          AND is_stale = FALSE
        RETURNING address_hash, normalized_address, provider_id
    ),
    queued_inserts AS (
        INSERT INTO cache_refresh_queue (
            address_hash, 
            normalized_address, 
            provider_id, 
            reason, 
            priority
        )
        SELECT 
            address_hash, 
            normalized_address, 
            provider_id, 
            'ttl_expired', 
            5
        FROM updated_cache
        -- ON CONFLICT DO NOTHING would go here if address_hash had a unique constraint in the queue
        RETURNING 1
    )
    SELECT COUNT(*) INTO invalidated_count FROM queued_inserts;

    RETURN invalidated_count;
END;
$$ LANGUAGE plpgsql;


-- ## SECTION 7: UPDATED_AT TRIGGER

-- WHY: Database-level triggers guarantee updated_at is always accurate, 
-- even if manual SQL updates are run outside the application ORM.
CREATE OR REPLACE FUNCTION trigger_set_timestamp()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER set_updated_at
BEFORE UPDATE ON property_cache
FOR EACH ROW
EXECUTE FUNCTION trigger_set_timestamp();

-- DEFINITIVE RECOMMENDATION: Option A (Range Partitioning)
-- ==========================================
-- **Justification**: For a **7**–**14** day TTL cache, Range Partitioning by `created_at` (or `expires_at`) 
-- is vastly superior <sup><span data-reference-id="072f2522-92f6-4f02-86ac-844b0a5c20a8">[7]</span></sup><sup><span data-reference-id="7352d421-600d-46b4-8acf-fcbe217dc749">[11]</span></sup><sup><span data-reference-id="92b707a0-2e37-4e89-8ffc-1912dec17348">[12]</span></sup>. 
-- 1. **TTL Management**: You can instantly reclaim disk space by dropping old partitions (e.g., `DROP TABLE pc_range_2026_02`) without generating massive WAL bloat or requiring heavy VACUUM operations, which is a major issue with DELETEs in Hash Partitioning <sup><span data-reference-id="b958cf98-0dc5-4929-ad57-4b959466463e">[10]</span></sup><sup><span data-reference-id="7352d421-600d-46b4-8acf-fcbe217dc749">[11]</span></sup>.
-- 2. **Write Velocity**: While Hash Partitioning prevents hot spots <sup><span data-reference-id="b4c3433a-b20e-415e-a14d-9ec0b5fecf41">[9]</span></sup>, a write velocity of **100**–**500** addresses per **45** seconds (~**11** writes/sec) is trivial for a single modern PostgreSQL partition. The "hot partition" problem only becomes a bottleneck at thousands of writes per second.
-- 3. **Query Pruning**: Queries filtering by date ranges will automatically skip scanning irrelevant partitions, drastically reducing I/O <sup><span data-reference-id="072f2522-92f6-4f02-86ac-844b0a5c20a8">[7]</span></sup><sup><span data-reference-id="aca830e2-1b50-4c2f-a6f8-277f89acff1f">[8]</span></sup>.

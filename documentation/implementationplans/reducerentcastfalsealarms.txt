educe RentCast False Alarms & Optimize BatchData Costs
This plan details how we will restructure the RentCast ingestion and BatchData validation pipeline to minimize false doors and achieve the lowest possible API spend. Currently, the system leaks too many Inactive RentCast listings into the validation queue, causing unnecessary credit burn.

Goal Description
Reduce RentCast False Alarms Locally: We will stop paying BatchData to tell us what we should already know. If an Inactive listing was taken off the market 120 days ago, it should have appeared in the Phase 1 Deed Records (which lag by 30-90 days). If it hasn't, it's an expired/withdrawn listing, not a sold home.
Implement Async Bulk Validation: For the truly ambiguous properties (recent Inactive listings that are still within the county lag window), we will send them to BatchData using the Async Bulk endpoint to optimize operational overhead and align with the cheapest architectural patterns.
User Review Required
IMPORTANT

Async Webhook Requirement: The BatchData Async Bulk endpoint requires a webhook URL to receive results. We will need to set up a new Base44 function (batchDataWebhookCallback) that is accessible via a public URL. Please confirm if Base44 supports exposing a public HTTP endpoint for this webhook.

Proposed Changes
RentCast Ingestion Filter (The "Lag Window" Rule)
This is the primary software-level cost saver. We will aggressively filter Phase 2 records before they even get marked as low confidence.

[MODIFY] processFetchChunk/entry.ts
Rule Changes for Phase 2 (listings_records):
We currently accept any Inactive listing removed after soldCutoff (up to 12 months ago).
New Rule: If a property has a removedDate older than 90 days (the maximum county deed lag), AND it does not already exist in the database from Phase 1 (deed_records), we must assume it is an Expired/Withdrawn listing and drop it.
Only properties with a removedDate within the last 0-90 days will be given sale_confidence = 'low' and queued for BatchData. Older false alarms will be assigned sale_confidence = 'REJECTED'.
BatchData Async Bulk Implementation
We will transition the validation queue from a 1-by-1 synchronous loop to a bulk async job.

[MODIFY] processValidationQueue/entry.ts
Change execution from loop to bulk slice: The function will pull up to 1,000 pending records from ValidationQueue.
API Call Update: Formulate the payload for POST https://api.batchdata.com/api/v1/property/lookup/async.
Options Object: Inject a webhookUrl (pointing to the new Base44 webhook endpoint) and requestId mapping.
Queue State: Update the queue items from pending to processing (awaiting webhook).
[NEW] batchDataWebhookCallback/entry.ts
Purpose: An endpoint to catch the POST payload from BatchData.
Logic:
Parse data.results.properties.
Evaluate listing.status and listing.statusCategory.
Upsert the results to PropertyValidationCache (sold or rejected).
Update the corresponding ValidationQueue items to completed.
If sold, update MasterProperty confidence to verified.

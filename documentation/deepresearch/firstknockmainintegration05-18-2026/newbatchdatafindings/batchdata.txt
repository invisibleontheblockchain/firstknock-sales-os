1. Current State
The v15 pipeline sends a bare POST to property/search with no options.datasets parameter:
body: JSON.stringify({ searchCriteria: { query } })
When the datasets array is omitted entirely, BatchData returns ALL datasets permitted by the API token's plan. This creates two distinct problems: uncontrolled data consumption from unrequested add-ons, and a missing owner name field that is not surfaced by either the basic or listing datasets.
If the plan includes the Listing Data add-on, listing.status, listing.soldPrice, and listing.soldDate ARE returned on the bare call
listing.statusCategory is read by v15 but is not explicitly documented in the BatchData schema — it may exist as an undocumented field or may always be empty/undefined
Owner name is NOT returned on a bare call unless the Property Owner Profile ("owner") add-on is included in the plan — the "basic" and "listing" datasets do not include owner name fields
The current plan includes Listing Data, so listing.status, listing.soldPrice, and listing.soldDate are being returned today

2. Cheapest Add-On Combination
Goal: Retrieve both (1) sold/deed status confirmation and (2) property owner name in a single property/search call, at the lowest possible incremental cost.
Option A — MLS-Based (Recommended for v15)
Preserves existing v15 field-reading logic. Requires only one new add-on.
"options": {  "datasets": ["basic", "listing", "owner"]}
"basic" — Base property data (address, building, lot). Cheapest base tier
"listing" — Provides listing.status, listing.soldPrice, listing.soldDate. Already in the plan; no new cost
"owner" — The only new add-on needed. Provides owner.fullName and owner.names[]
Why this is cheapest: "listing" is already paid for — adding it explicitly to datasets doesn't add cost, it just scopes the response. "owner" is the single new add-on required to surface owner name. No other dataset is needed to satisfy both goals.
Option B — Deed-Based (Recommended for v16)
Swaps the listing add-on for deed, providing county-recorded sale data as ground-truth confirmation of ownership transfer.
"options": {  "datasets": ["basic", "deed", "owner"]}
Provides deedHistory[].recordingDate, sale.lastSale.saleDate, and sale.lastSale.price
Advantage: Deed recording is ground-truth confirmation of ownership transfer — eliminates false positives from properties in escrow (30–60 day MLS-to-deed gap)
Disadvantage: Requires updating v15 field-reading logic; listing.status, listing.soldPrice, and listing.soldDate would no longer be populated
Cost: Replaces "listing" add-on with "deed" add-on — net cost depends on relative add-on pricing (contact BatchData sales)

5. Cost Impact
BatchData charges per-record: each property returned equals one billable record against the monthly quota. Add-on datasets are monthly line items on top of the base plan. Critically, adding the "owner" dataset does not increase the number of billable records — it enriches each existing record with owner data.

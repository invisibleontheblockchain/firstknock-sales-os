absent from /listings/sale, /listings/rental, /properties, and every other endpoint. Any code

assumption — it is confirmed.

The soldPrice field does not exist on the /listings/sale endpoint or any other listing endpoint. The

recording practices. There is no real-time sold price available through RentCast.

The history object exists and was added in September 2024. However, its utility for disposition
classification is severely limited. The event field within each history entry only records whether the
historical event was a Sale Listing or Rental Listing — it does not record outcomes such as Sold,
Expired, Withdrawn, or Cancelled. The history object records listing events, not listing outcomes.
Each entry contains: event, price, listingType, listedDate, removedDate, and daysOnMarket.
Important Distinction: The history object tells you WHEN a property was listed and for HOW LONG
— it does not tell you WHY the listing was removed. A listing removed after 25 days and a listing
removed after 365 days both appear as identical "Sale Listing" events in the history object.

Five classification strategies are identified, ranked by estimated precision. Each strategy uses only
fields available on existing RentCast endpoints at no additional API cost. Precision estimates are
based on known MLS behavioral patterns and RentCast data characteristics — empirical validation
against a labeled dataset is recommended before production deployment.

5. Recommended Zero-Cost Pipeline
The following pipeline combines all five strategies into a single decision workflow. The pipeline is
designed to maximize zero-cost classification coverage while routing only genuinely ambiguous
records to BatchData. The pipeline processes each Inactive listing record in three sequential steps.
5.1 Step 1 — Hard Signal Check (Binary Decision)
Before applying any heuristic scoring, perform a hard lookup against the /properties endpoint to
check for a confirmed deed-recorded sale. This step resolves the highest-confidence cases
immediately and removes them from the scoring pipeline.
1. Call GET /v1/properties?address={formattedAddress}&saleDateRange=90
2. Parse the response for the lastSaleDate field
3. If lastSaleDate is present and falls within the past 90 days rightarrow classify as SOLD
immediately. Do not proceed to Step 2.
4. If lastSaleDate is absent, null, or older than 90 days rightarrow proceed to Step 2 (heuristic
scoring)
5. If the /properties call returns no record rightarrow proceed to Step 2 (heuristic scoring)
Optimization Tip: For high-volume processing, consider using saleDateRange=0:45 (min:max
syntax, available since August 2025) to target only very recent sales and reduce false positives
from properties that sold long ago but recently re-listed.
5.2 Step 2 — Heuristic Scoring Model
For records that did not resolve in Step 1, apply the following point-based scoring model. Each
signal contributes a positive score (sold indicators) or negative score (expired/withdrawn
indicators). Compute the total score across all applicable signals.

6.2 statusCategory as Query Filter
statusCategory does not exist as a query parameter on any RentCast endpoint. It cannot be used
to filter listings by disposition type. Any query string containing statusCategory will be silently
ignored by the API, which will return results based on the default status=Active filter. This is a
silent failure mode — no error will be returned, but the results will be incorrect.

7. Open Questions & Limitations
The following items represent unresolved questions or known limitations that could affect the
accuracy of the zero-cost classification pipeline. Each item should be investigated before
production deployment, either through empirical testing against labeled data or direct inquiry with
RentCast support.
1. price Field Behavior on Inactive Listings — Does the price field update to the sold price when
a listing transitions to Inactive status? Documentation states that Inactive listings reflect "last
known listing attributes" — suggesting the list price is preserved, not updated to the sale
price. However, this has not been empirically verified. If price does update to sold price on
Built with Spine AI on 2026-03-30 15
some records, it could serve as an additional classification signal (price significantly below list price may indicate a distressed
sale or price reduction before expiration).
2. daysOnMarket Calculation Methodology — How RentCast calculates daysOnMarket is not
documented. It is unclear whether DOM is counted from listedDate, from the first MLS
appearance date, from createdDate, or from another reference point. This ambiguity directly
affects the precision of Strategy 2 (DOM threshold heuristics). A discrepancy of even 5–7
days in DOM calculation could cause misclassification at the 90/180/365-day boundary
thresholds.
3. RentCast Data Refresh Cadence — The reliability of the lastSeenDate signal (Strategy 4)
depends entirely on how frequently RentCast refreshes listing data from MLS sources. If
RentCast updates listings daily, the lastSeenDate gap is a meaningful signal. If updates
occur weekly or less frequently, the gap may reflect data collection timing rather than true
listing behavior. RentCast does not publicly document its update frequency.
4. history Object Completeness and Truncation — It is unknown whether the history object is
truncated for older listings or whether all historical listing events are preserved indefinitely. If
history is truncated (e.g., only the last 12 months of events are retained), the multi-entry
pattern signal in Strategy 5 will be unreliable for properties with older listing histories. This is
particularly relevant for properties in slow-moving markets that have been listed and relisted
over multiple years.
5. County-Specific Deed Recording Lag — The 30–90 day lag on lastSaleDate via /properties
varies significantly by county. High-volume counties such as Cook County (IL), Los Angeles
County (CA), and Harris County (TX) may have lags at the longer end of the range, reducing
Strategy 1 recall in those markets. A county-specific lag lookup table would improve pipeline
accuracy for high-volume markets. Consider building a lag calibration dataset by comparing
lastSaleDate timestamps against known sale dates in a sample of markets.
6. Behavior of saleDateRange with Value 0 — The saleDateRange parameter has a
documented minimum of 1 day. The behavior when saleDateRange=0 is provided is not
documented. The min:max range syntax (e.g., saleDateRange=0:45) was added in August
2025 — it is unclear whether the lower bound of 0 is valid or whether it defaults to 1. This
should be tested before using the 0:45 range syntax in production.
7. MLS Coverage Gaps — RentCast's listing data is sourced from MLS feeds, but MLS
coverage is not universal. Some markets, particularly rural areas and states with lower MLS
adoption, may have incomplete listing data. In these markets, the history object may be
sparse or absent, and the lastSeenDate signal may be less reliable. The classification
pipeline should include a data quality check to flag records with missing or null values for key
signal fields.

4. Failure Modes:





Address Formatting: Discrepancies between MLS entry and county records (e.g., “Street” vs. “St”, or missing directional prefixes).



Unit Numbers: Variations in condo/apartment formatting (e.g., “Unit A” vs. “#A").



Parcel vs. Postal: Phase 1 may use legal parcel addresses, while Phase 2 uses postal addresses.

5. Implementation Complexity: Medium
While building a hash set is computationally trivial (Low complexity), implementing robust address normalization and fuzzy matching to prevent false negatives elevates the complexity to Medium.



Concrete Estimate: ~21% overlap, meaning ~45 of 215 Phase 2 records are already confirmed by Phase 1, reducing BatchData calls by that exact amount.



STRATEGY 2: Freshness Window Filtering

Mechanism: Only send Phase 2 Inactive listings to BatchData if their removedDate falls within the last 0–30 days.

1. Logic & Assumptions:
The logic assumes that if a listing was removed 31–90 days ago and successfully sold, the county deed recording lag (30–90 days) has passed, and it will appear in Phase 1 [1]. If it is not in Phase 1 after 30 days, it is assumed to be expired or withdrawn. This assumes county recording never exceeds 90 days and that Phase 1 data is perfectly synced with county updates.

2. Distribution of Inactive Listings:
In a buyer’s market, delistings are high, with 5.5% to 6% of inventory removed monthly [3][4]. The typical delisted home sits on the market for ~100 days before removal [6]. Because the 215 records represent an accumulated pool, older records dominate.





0–30 days: ~20% (~43 records) - Recent removals, highly active.



31–90 days: ~35% (~75 records) - Aging out of the lag window.



>90 days: ~45% (~97 records) - Stale inactive listings, almost certainly expired/withdrawn [4][6].

3. Expected BatchData Call Reduction:
If only the 0–30 day bucket requires validation, you only send ~43 records to BatchData.





Reduction: 215 - 43 = 172 calls avoided.

4. Risk of False Negatives:
High. Properties that sold 35 days ago but are experiencing a 60-day county recording lag will be skipped by BatchData and missed entirely by Phase 1, resulting in missed door-knocking opportunities.

5. Edge Cases:
Foreclosures, short sales, estate sales, and delayed closings frequently break the 30–90 day assumption, often taking 120+ days to record legally.



Concrete Estimate: Freshness Window alone reduces BatchData calls from 215 to approximately 43 per area pull.



STRATEGY 3: Combined Strategy (Cross-Reference + Freshness)

Mechanism: Cross-reference against Phase 1 first, then apply the Freshness Window to the remainder.

1. Combined Reduction Calculation:





Start: 215 Phase 2 records.



Apply Cross-Reference: Removes ~45 verified sold records. Remaining = 170.



Apply Freshness Window: Of the remaining 170, only validate those with a removedDate of 0–30 days. Since Phase 1 mostly catches older sales (due to lag), the 0–30 day bucket remains largely intact at ~40 records.



Final BatchData validation volume: ~40 records.

2. Combined Reduction Percentage:
(215 - 40) / 215 = 81.3% reduction vs. the baseline.

3. Estimated False Negative Rate:
~10-15%. Genuine sales that occurred 31–90 days ago but are stuck in a prolonged county recording lag will be skipped by the Freshness Window and won’t yet be in the Phase 1 hash.

4. Estimated False Positive Rate:
Moderate. You are still paying to validate ~40 recent removals, many of which are simply withdrawn (like the 117 Phillips St example).

5. Sequencing Advantage:
Cross-referencing first is critical. If you apply the Freshness Window first, you might discard a 45-day-old removal that did successfully record in Phase 1, losing the verified lead.



Concrete Estimate: Combined strategy reduces BatchData calls from 215 to approximately 40 per area, an 81% reduction.



STRATEGY 4: Delta Validation (Weekly Batch Frequency)

Mechanism: Validate only NEW Phase 2 entries not present in the previous week’s pull.

1. Weekly New Inactive Listings:
Inactive listings are a terminal status; they do not cycle back to Active without a new MLS ID [8]. Therefore, the delta consists purely of newly removed listings.

2. Estimate of Weekly Additions:
If an area sees ~50 sales per 90 days (~4 sales/week), and we assume a ~30% sold rate for inactive listings [1], the total new inactive listings (sold + expired + withdrawn) would be roughly 13 per week per ZIP code.

3. BatchData Call Reduction:
Instead of re-validating the static pool of 215 records every fetch, you only validate the ~13 new records.





Reduction: 215 - 13 = 202 calls avoided per week.

4. Infrastructure Required:





Persistent database (PostgreSQL/Redis) to store previously seen listingIds.



Deduplication logic during the ingestion phase.



Cron jobs for weekly batch processing.

5. Interaction with Freshness Window:
They are highly complementary, bordering on redundant. If you only process weekly deltas, every record you process is, by definition, 0–7 days old, naturally fitting inside the 0–30 day Freshness Window.



Concrete Estimate: Delta validation reduces weekly BatchData calls to approximately 13 new records/week vs. 215 if re-validating all records.



STRATEGY 5: Confidence Scoring Without BatchData

Mechanism: Build a scoring model using RentCast fields to assign a “likely sold” probability.

1. Predictive Power of RentCast Fields:





Short DOM (<14 days): Ambiguous. It indicates either high demand/priced right (fast sale) [7][9], or a rapid withdrawal due to seller remorse/issues (e.g., 117 Phillips St, withdrawn in 10 days).



Long DOM (>60 days): Highly predictive of expired/withdrawn. Homes sitting 60+ days are considered “stale” and make up 70% of delistings [4][6].



Listing Duration (removedDate - listedDate): If duration equals DOM + 30 to 45 days (standard escrow period), it is highly likely a successful sale. If duration equals DOM exactly, it was likely withdrawn/canceled.

2. Simple Scoring Rubric:





Base Score: 0



DOM > 60 days: -50 points (Likely stale/withdrawn) [6]



Duration = DOM + 30-45 days: +80 points (Standard escrow signature)



Price reduced before removal: -20 points (Struggling listing) [5]

3. Precision/Recall Tradeoff:
To achieve >90% precision (avoiding false doors like 117 Phillips St), you must set the threshold very high (e.g., requiring the exact escrow duration signature). This sacrifices recall, missing cash sales (short escrow) or delayed closings (long escrow).

4. Threshold & Classification:
At a threshold of +75 points, only about 15% of the 215 Inactive listings would be classified as “likely sold” with >90% precision.

5. Limitations:
RentCast lacks contract status updates, MLS agent remarks, and final sale price confirmation. It cannot definitively distinguish a 10-day cash sale from a 10-day withdrawal without BatchData [2].

6. Viability:
It is not viable as a standalone replacement due to the high risk of false doors (wasting sales reps’ time). It is best used as a pre-filter to skip BatchData for obvious expirations (DOM >90).



Concrete Estimate: Confidence scoring alone can eliminate BatchData for approximately 45% of records (the obvious expirations) with >90% precision, but requires BatchData for the remaining 55%.
